Building a Custom Blogging Platform

I've tried blogging several times over the years using various platforms like Medium and WordPress, as well as more developer focused software like Jekyll. Each time, I eventually stopped writing, frustrated by clunky interfaces, limited control over the publishing workflow, and platforms that felt designed for everyone except how I actually work.

So when I decided to start writing again, I took a different approach: I built my own blogging platform from scratch. I call it MonoBlogger.

Why Build Your Own?

The decision to build MonoBlogger wasn't driven by technical complexity for its own sake. I had specific requirements that existing platforms couldn't meet well. I write primarily in plain text editors with markdown, and having to paste into web editors or deal with formatting quirks breaks my flow. I also wanted my posts to exist as files on my computer rather than just in a database somewhere, making backup, version control, and migration trivial.

Academic and technical writing often involves images, diagrams, and data files, so I wanted each post to live in its own folder with all related assets together. The publishing process needed to flow directly from writing to deployment without manual steps or complex workflows. Finally, I wanted complete control over the design to create a specific aesthetic that reflected my interests in formal systems and terminal interfaces.

MonoBlogger's Architecture

MonoBlogger is a static site generator built in Node.js that transforms markdown files into a complete blog. The architecture follows a modular design that separates concerns cleanly:

Content Organisation: Each post lives in its own folder under a posts/ directory, containing an index.md file and any associated images or assets. Pages like About live separately in a pages/ folder for clean organisation.

Modular Codebase: Rather than a monolithic script, MonoBlogger uses separate modules for configuration loading, CSS generation, feed creation, HTML templating, content processing, and file operations. This makes the codebase maintainable and extensible.

Build Process: The build system reads all markdown files, parses frontmatter for metadata, converts relative image paths to absolute ones automatically, and generates a complete static website using the marked library for markdown processing.

Theme System: CSS is generated programmatically with configurable theme variables, creating MonoBlogger's signature terminal-inspired design with monospace fonts, solid borders, and a carefully chosen dark color palette.

Design Philosophy

The visual design draws from terminal interfaces and ASCII art traditions, which is why I named it MonoBlogger. The name comes from the monospace typography that defines its aesthetic and the focused, single-purpose nature of the platform.

Monospace fonts (Monaco, Menlo, Ubuntu Mono) for headers and UI elements give it a computational feel, while sans-serif fonts for body text maintain readability. The dark theme with subtle borders creates clear information hierarchy without unnecessary decoration. Hash marks (#) prefix headers, arrows (>) indicate navigation links, and brackets frame category tags give that nice ASCII aesthetics to the design.

Category filtering allows readers to focus on specific topics, while the folder-based post organisation makes it easy to manage posts with multiple assets. Future post scheduling lets me write content in advance and have it publish automatically when the date arrives.

Technical Features

MonoBlogger includes several features that emerged from real-world usage:

Asset Management: Images and files stored alongside posts get automatically copied to the correct public directories during build. Relative paths in markdown (./image.jpg) automatically convert to absolute paths (/images/post-slug/image.jpg).

Dual Feed Support: Automatically generates both RSS 2.0 and Atom 1.0 feeds with proper XML formatting and absolute URLs for maximum compatibility.

GitHub Pages Integration: One-command deployment that builds the site and pushes to GitHub Pages, with separate repositories for source code and published content.

Responsive Design: Terminal aesthetic adapts well to mobile devices with horizontal scrolling navigation and optimised typography scaling.

What I Learned?

Building MonoBlogger taught me about my actual writing habits rather than what I think they should be. I discovered that I write in bursts and benefit from being able to prepare multiple posts when inspired, then schedule them for regular publication.

The folder-based organisation proved essential for posts involving technical content. Being able to keep screenshots, diagrams, and reference materials alongside the post text eliminates the friction of managing assets separately. Most importantly, having full control over the publishing stack removes barriers to writing.

Open Source Release

I've open sourced MonoBlogger so that others can benefit from it as well. The modular architecture makes it easy for others to modify and extend. The repository contains only the generator code with no personal content, making it immediately usable for anyone who wants a text-based blogging platform.

Key benefits for users:

  • MIT licensed for maximum flexibility
  • Minimal dependencies (just marked for markdown processing)
  • Clear documentation and examples
  • Folder-based content organisation
  • Configurable themes and styling
  • Automatic deployment to GitHub Pages

The Trade-offs

Custom platforms come with obvious tradeoffs. There's no comment system, no built-in analytics, and no social features that platforms like Medium provide. The audience discovery that comes with established platforms is entirely absent, meaning building readership requires more deliberate effort.

Maintenance becomes your responsibility. When you want new features, you have to build them. When something breaks, you have to fix it. For someone who enjoys technical projects, this is more feature than bug, but it represents a real ongoing commitment.

MonoBlogger reflects my specific writing patterns and preferences, working well for technical and academic content with images and references but potentially being overkill for simpler blogging needs.

Why I Love It?

MonoBlogger succeeds because it's designed around actual workflows rather than trying to accommodate everyone's needs. The build process is fast, the design loads quickly, and the deployment is reliable. More importantly, the writing experience feels frictionless.

There's something satisfying about running a terminal command and seeing your blog being updated almost in real time.

Was It Worth It?

Building a custom blogging platform makes sense if you have specific workflow requirements that existing platforms don't meet well, enjoy technical projects, and want complete control over your content and presentation.

MonoBlogger might be a good middle ground. It provides the flexibility and control of a custom solution but with significantly less implementation work than building from scratch. The terminal aesthetic won't appeal to everyone, but for developers and researchers who spend their days in code editors and terminals, it creates a familiar, comfortable writing environment.

The platform doesn't make sense if you want to focus purely on writing without technical concerns, need built-in social features, or prefer the audience discovery that established platforms provide.

For me, MonoBlogger was worth building. The platform enables rather than hinders my writing, building it taught me about both my technical preferences and my writing habits, and releasing it as open source means others can benefit from the work too.

Sometimes the best tool is the one you make yourself, and then share with others who might need the same solution.
